## 0. THE HEADLINE — the framing was wrong, and the correction changes the plan

The 20-agent audit said: _"the backfill reconciles GREEN while destroying data."_ True, but it is the second-order finding. The first-order finding is:

> **The build-v3 DDL cannot run the product.** It was written without reading the shipped backend. The backfill is downstream of that, and `reconcile_v3.sql` is downstream of both.

Ten of the twenty-two ledger rows below are _"the target is missing a column the code on this branch already writes."_ `runtime.pairing` is missing 9 columns. `runtime.conversation_turn` and `tenant.birthday_reward` do not exist at all and are queried unconditionally on the scan path. `tenant.contact` has no unique index, so `identity.resolver.ts`'s `ON CONFLICT` cannot compile (SQLSTATE 42P10). `runtime.outbox_event` lost the `UNIQUE(idempotency_key)` that _is_ the bot's exactly-once lock.

So the primary artifact is a **DDL revision**, and the single highest-value control is not a reconcile tier — it is a **column-level schema-parity test**. Today's gate (`schema-parity.integration.ts`) matches `from|join|into|update <schema>.<name>` and asserts the _relation_ exists. That is exactly why nine missing columns on `runtime.pairing` sit in `backend-convergence-map.md:35` filed as **"Already correct (no change)."** Extend it to assert every column named in every SQL string exists in the target, and it catches ten of the twelve destructions **before a single row moves**.

### Corrections to the brief itself (both load-bearing, both re-run)

**C1 — "STAMPS DRIFT" IS FALSE. There is no drift. There never was.**

```
SUM(coalesce((metadata->>'seals')::int, 1)) over loyalty.visit_events = 624
SUM(loyalty.cards.total_visits)                                       = 624
cards where the two disagree                                          =   0   (of 445)
raw row count of visit_events                                         = 537
```

A `visit_event` is a **scan**. `total_visits` counts **stamps**. One scan carries up to 50 (Kalala bulk-seal, commit `1ee32aa`, PR #39 — and `git merge-base --is-ancestor 1ee32aa HEAD` is **false**, so the backfill author was reading a tree in which a scan could only ever be one stamp). 28 bulk-seal events credit 115 stamps across 22 cards; **115 − 28 = 87**. The entire "drift" is a units error.

Consequences: the 87 `generate_series` rows fabricate history to close a hole that does not exist — **delete them**. And the "rigorous" alternative floated in review (_elect `visit_events` as the authority, carry only 537, log the 87 as loss_) would **delete 87 real stamps from 18 real Kalala customers whose Google Wallet passes show them today**. It is the single most destructive proposal anyone made. Killed.

The real defect is that `tenant.loyalty_visit` has no magnitude column. **A row is not a magnitude.**

**C2 — the raw phone was never discarded.** `core.contact_methods.display_value` is populated 458/458 and still holds `+14804016182`. Only `normalized_value` is corrupted. The `+52 480` corruption is **fully repairable from data we already hold in both databases**.

## 1. THE METHODOLOGY

### 1.1 What we adopt: **snapshot → rebuild-in-one-transaction → assert → flip**

The mechanics question ("dual-write? CDC? FDW?") **collapses because downtime is allowed and the data is tiny.** Measured, not asserted:

|                                      | measured                                        |
| ------------------------------------ | ----------------------------------------------- |
| migration surface (8 domain schemas) | **16,373 rows** (26,051 incl. `observability`)  |
| `pg_dump -Fc` of all 8 schemas       | **0.98 s wall, 7,715,015 bytes**                |
| whole DB                             | 54 MB                                           |
| sequences in any domain schema       | **none** (all-UUID PKs → no sequence sync)      |
| `pg_cron`                            | **not installed**                               |
| write rate, last 30 days             | ledger 23 · visits 179 · orders 2 · messages 54 |

A one-hour freeze costs ≈ 0 rows. Every zero-downtime technique in the literature buys uptime or throughput. We need neither.

**The runner we already have is the right architecture and nobody noticed.** `00_run_backfill.sh:29-30` does `drop database if exists $DB; create database $DB template $TEMPLATE`. The source is never mutated; the target is a pure rebuild. That is snapshot-rebuild, for free. **The stated prod plan (postgres_fdw replay) throws it away for something strictly worse.** Port the runner; kill the FDW plan.

**Cutover shape:**

1. **Freeze** — 3 switches, not N: BullMQ schedulers (`lifecycle.scheduler.ts`, `leads.scheduler.ts`), the Twilio inbound webhook, Google Wallet callbacks → umi-cash. (`apps/umi-cash/vercel.json` is `{}` — no Vercel crons. No `pg_cron`.)
2. **Freeze proof** — _time-based_: assert no `queue.inbound_events` received in the last N minutes. **Do NOT** assert `status IN ('accepted','processing') = 0` — measured: **all 395 inbound rows are `status='accepted'` forever**, including the newest (2026-07-08). The status is never advanced. A gate that is red on a healthy system is a gate that gets waived at 4am.
3. **Signed snapshot** — `pg_dump -Fc` of the 9 source schemas (0.98 s, 7.7 MB) + `sha256` + retention date + named PII + named restorer. **This one artifact is what makes every `reversible: true` in §3 a true statement.** Until it has an owner, they are all lies.
4. **Rebuild** — one `psql --single-transaction -v ON_ERROR_STOP=1`, DDL + staged loads + assertions, all inside one transaction. **A red assertion ROLLBACKs the entire build.**
5. **Flip** — repoint apps. Maintenance page for both (umi-cash on Vercel and umi-api on GHCR→VPS do not deploy atomically; take the skew as downtime, not as compatibility machinery).
6. **Make the old world unwritable, do NOT drop it** — `REVOKE INSERT/UPDATE/DELETE` on `core/loyalty/ops/comms/grow/device/kitchen`. Keep read-only for weeks, until the new world has survived a reward redemption, a Wallet push, a WhatsApp order and a KDS ticket. `DROP SCHEMA loyalty CASCADE` in the cutover window is the one step that makes failure unrecoverable.
7. **Accept the rollback boundary out loud.** No reverse replication. At ~0.8 ledger rows/day and 2 orders/month, discarding an hour of new writes discards ≈0 rows, all reconstructible from Twilio/Wallet logs. Say it; don't pretend rollback is free forever.

### 1.3 What we REJECT, and why

| Rejected                                                                        | Kill                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`postgres_fdw` replay** (the current stated prod plan)                        | Three independent kills. (1) **No two-phase commit** — PostgreSQL docs F.38.4: _"it is currently not supported by postgres_fdw to prepare the remote transaction for two-phase commit."_ A cross-server replay is **not atomic**; it can leave prod half-migrated with no rollback. (2) `batch_size` defaults to **1** — one round-trip per row, for zero benefit over a 0.98-second dump. (3) It **embeds the source password in the target**, violating our own `SECURITY_GATE.md` **D8**. Same server ⇒ coexist, one local transaction. Different server ⇒ `pg_dump`/`pg_restore` first, then everything is local. |
| **Dual-write (Stripe four-phase)**                                              | Buys zero-downtime, which we declined to buy. It means a **third implementation of the mapping**, in TypeScript, across two apps — and you _still_ need the backfill and the comparison harness. | **CDC / logical replication / Debezium**                                        | _"The database schema and DDL commands are not replicated."_ It moves rows between **identically-shaped** relations. ## 2. THE DECISION PROCEDURE

For any fact that does not translate. Short enough to actually use.

**Step 1 — Establish the facts, from the DB and from grep. Never from the schema's intent.**

- `SELECT count(*)`, `SELECT DISTINCT`, and the per-row shape of the source column.
- `grep` for every reader and every writer. **A column with no reader is not a fact; it is sediment.**
- Ask: is this a **cache** (recomputable) or an **authority** (nothing else knows it)? _Run the derivation._ If it does not reproduce the column, it is not a cache. (`EGR-7232925905`: `total_visits=7, visits_this_cycle=7, pending_rewards=0` under a 7-stamp rule — no modular derivation reproduces it, because it accrued under the old 10-stamp rule. That one row proves `pending_rewards` is an independent source.)

**Step 2 — Judge against three constituencies.**

1. **THE FINAL USER** — what the café customer and owner can _see and rely on_: Saldo, stamps, a wallet pass already on their phone, order history, the bot replying, **and the owner's ability to see the liability they owe.**
2. **THE DEV TEAM** — legibility, testability, one reason to change. A boolean must not become a substring.
3. **THE ENTERPRISE TRUTH** — does the model faithfully represent what is true? An entitlement that can expire is not a rule. A disabled café is not active. A cancelled line is not a note.

**Step 3 — Tie-breaks, in strict order.**

> **T1. User-visible irreversible harm beats everything.** Money, stamps, a live wallet pass, order history, the bot replying, seeing your own liability. If an option harms these, it loses regardless of how clean it is.
>
> **T2. Truth beats dev convenience.** A concat is convenient. It is a lie. Truth wins.
>
> **T3. Dev legibility beats truth-completeness when the extra truth has no consumer.** Carrying a fact nobody reads harms the dev team _and_ clutters the truth. **Delete it.** (This is where KISS is earned, and it is why we delete 3,401 turn rows, 417 outbox rows, 9,858 observability rows and the entire birthday feature.)
>
> **T4. Burden of proof is asymmetric.** `DELETE` requires a **grep proving zero consumers plus a row count**. `CARRY` requires a **named consumer**. If you can prove neither, carry it and open a ticket — but write the ticket.
>
> **T5. Capacity beats data.** A missing row is replayable from the snapshot. A **missing column** means the fact can never be captured going forward, and no later migration can reconstruct it — you cannot go back and ask a barista which drink she voided in March. **When in doubt, ship the column and drop the rows.**
>
> **T6. When truth demands a redesign, ship the truthful CAPACITY (DDL) at cutover and the truthful FEATURE (code) after.** Never build a net-new feature on cutover night, and never on the money path. This single rule settles both the reward-entitlement case and the birthday case, and it is why we reject the full entitlement-ledger redesign _at cutover_.

**Step 4 — Write it down.** One row in `column-ledger.tsv`, one row in §3. `DROPPED` rows carry the grep that proved it.

## 4. THE VERIFICATION CONTRACT

### 4.1 The current gate is not weak. It **cannot fail.**

`reconcile_v3.sql` contains **zero** `raise`, zero `do $$`, zero assertions. It is 5 bare `SELECT`s. And it is the **only** psql invocation in `00_run_backfill.sh` without `-v ON_ERROR_STOP=1` (line 49).

Proven twice:

- I opened a transaction, deleted `order_event`, `order_item`, `customer_order`, all 136 products, **all 624 stamps**, **all 417 wallet passes**, all 398 pass devices, and set every café to `'suspended'`, then ran it. **Byte-identical output to the healthy run. Exit 0.**
- In an earlier run, **every single reconcile query threw** `ERROR: current transaction is aborted`. `psql` still exited **0**, and the runner still printed `done:`.

Coverage, counted: **38 populated target tables. It count-checks 9.** It has zero coverage of `loyalty_visit` (the stamps), `loyalty_wallet_pass`, `pass_device`, `product`, `order_item`, `order_event`, `loyalty_redemption`, `staff`, `device`, `session`. Its "orphan sweep" is **vacuous by construction** — all six FKs are DDL-enforced; Postgres already rejected the insert. Reporting that as proof is the reasoning error that produced the current false confidence.

**Root cause, named:** every zero-downtime tool in the industry verifies **IDENTITY**. Only the two teams who actually changed the _model_ verified **SEMANTICS**. A row count is a necessary condition and nothing else — `order_item` is 73→73 **green** while `is_cancelled` is destroyed; `order_event` is 78→78 **green** because the count is taken _after_ the filter.

### 4.2 What replaces it: **one file that raises**, in seven families

Delete `reconcile_v3.sql`. Write `assertions.sql`: plain `do $$ … raise exception`, run **inside the migration transaction**, with `ON_ERROR_STOP=1`. **A red assertion ROLLBACKs the build.**

**No `runtime.reconcile_finding` table. No `assert()` procedure. No `BLOCKER/WARN/INFO` severity.** Two permanent objects in the sealed `runtime` schema — whose own header says it holds _"nothing that is a business fact"_ — to hold one-shot migration bookkeeping. And **a `WARN` is a check that has decided not to decide.** Every check aborts the cutover, or it is not a check.

## 5. HOW THE BACKFILL CODE IS ORGANIZED

```
docs/migration/build-v3/
  10_umi.sql  20_tenant.sql  30_runtime.sql  90_rls.sql     # DDL (the primary artifact of this PR)
  column-ledger.tsv                                          # F1. one row per source column. in git.
  backfill/
    00_run.sh          # ONE psql --single-transaction -v ON_ERROR_STOP=1
    10_stage.sql       # stg.* views: pure, 1:1, rename+cast+total CASE. no joins, no aggregates,
                       #   no now(), no gen_random_uuid(), no generate_series.
    20_load.sql        # dumb loaders: insert into <target> select * from stg.x ...; FK-ordered.
    30_assert.sql      # F1–F7. raises. INSIDE the transaction. red => ROLLBACK.
    40_darkread.sql    # F8. old-vs-new answer diff.
```

**Ordering rules** (load-bearing, keep them): the loyalty vertical first (it creates `tenant.business` and `tenant.customer`); cross-schema FKs and RLS applied **after** the data lands. The existing `00_run_backfill.sh:11-19` documents this correctly and it was hard-won.

**Idempotency mechanism.** Declared: **the backfill is NOT resumable. On failure, rebuild from template.** One transaction is the whole enforcement. No watermark table, no resume ledger — that is resume-driven engineering for a **0.98-second** rebuild. But `ON CONFLICT` semantics still get declared **per table**, because the two forms are not interchangeable:

- **Append-only ledgers** (`loyalty_stored_value_ledger`, `loyalty_visit`, `order_event`, `message`, `audit_log`) → `on conflict (id) do nothing`, arbitrating on `idempotency_key` where one exists. Converges to _"a row exists"_ — correct for an immutable fact.
- **Mutable projections** (`business`, `customer`, `product`, `loyalty_card`, `branch`) → `on conflict (id) do update set …`. Converges to _"the **right** row exists"_ — so a repair-replay **fixes** drift instead of cementing it. `DO NOTHING` on a mutable projection is a trap: a half-written row survives the sweep.
- **Never** sprinkle `ON CONFLICT DO NOTHING` as a reflex. Where two source rows collide on one target key it converts a loud crash into a **silent shortfall you cannot explain** — the same disease as the reconcile: the appearance of a control with none of the substance. It is safe **only** paired with a preflight source-side uniqueness assertion.

**Deterministic ids.** Every insert that today relies on a `gen_random_uuid()` default gets a derived id: `uuid_generate_v5(<committed namespace>, <source key>)`. Verified exact for `product_option_group` (**66 groups / 66 products, 1:1**). Pin the seed tables (`umi.channel_type`, `umi.feature`, `umi.plan`) to **fixed literal UUIDs** — then `backfill_comms.sql:43`'s `-- look up by key (UUIDs are random per build)` becomes a constant, and golden-file testing becomes possible for the first time.
