# Issue tracker: Trello (+ GitHub PRs as the review surface)

Issues and PRDs for this repo live as **Trello cards**; code review happens on **GitHub PRs**. A card is the *spec*; a PR is the *implementation + review*. Use the official **Trello MCP** (`https://mcp.trello.com/v1`) for card operations, and the `gh` CLI for pull requests. One workspace per MCP connection. If the Trello MCP is not connected in the current session, say so and stop; do not guess a card's contents.

Cards and PRs are independent: a card can exist with no PR (future work, a bug for another dev, a client request) and a PR with no card (a small fix). A card may be created by the ticket skills or by a person straight on the board — both are valid; nothing here assumes a card came from an agent.

## Where the spec lives

The linked Trello card: **description** = requirement, **checklist items** = acceptance criteria, **comments** = clarifications. **List** = status; **labels** = type / area.

## Linking a PR to its card — optional

Opportunistic, never required — most PRs won't carry one. When a PR does implement a specific card, it can declare it with a line in the PR body:

```
Trello: https://trello.com/c/<shortLink>
```

Resolve the card shortLink in this order: (1) a `Trello:` line in the PR body; (2) any `trello.com/c/<shortLink>` URL in the PR body or latest commit trailer; (3) a shortLink embedded in the branch name. If none resolve, that is normal — `code-review` runs its Standards axis only and does **not** treat a missing link as a finding.

## Conventions (via the Trello MCP)

- **Fetch a card** — get by shortLink or id, then read its description, checklists, and comments.
- **Search** — search cards across the connected workspace by text, list, or label.
- **Create a card** — create in the agreed intake list with a title, a structured description, and a checklist of acceptance criteria.
- **Comment / label / status** — comment on a card, apply a label, or move it to the target list.

Discover the exact `mcp__trello__*` tool names at runtime; the operations above map to whatever the connected server exposes (boards, lists, cards, checklists, search).

## When a skill says "publish to the issue tracker"

Create a Trello card, or comment on the relevant one, via the Trello MCP. For AI-authored triage output, start the card or comment body with `> This was generated by AI.` Cards a person creates by hand (e.g. from a client message) are ordinary cards and need no disclaimer.

## When a skill says "fetch the relevant ticket"

Resolve the card shortLink (order above) and read the card via the Trello MCP — description, checklists, and comments are the full ticket.

## PRs as a triage surface

**PRs as a request surface: no.** PRs are the review surface for internal changes, not an external feature-request queue. `code-review`'s Spec axis reviews a PR's code against its linked Trello card; `triage` operates on Trello cards.

## Wayfinding operations

Used by `wayfinder`. The **map** is a single Trello card labelled `wayfinder:map`; **child tickets** are cards linked to it, labelled `wayfinder:<type>` (`research` / `prototype` / `grilling` / `task`), assigned to the driving dev once claimed. **Blocking**: a `Blocked by: <shortLink>, <shortLink>` line at the top of the child card's description (Trello has no native dependency graph); unblocked when every blocker card sits in Done. **Frontier**: the map's open child cards with no unmet `Blocked by` and no assignee, first in map order. **Claim**: assign the card to yourself. **Resolve**: comment the answer, move to Done, append a pointer to the map's Decisions-so-far.
